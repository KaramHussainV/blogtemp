Migration From
On-Premises to AWS Cloud

A comprehensive guide to successfully migrating your infrastructure to the cloud

April 17, 2024 By Karam Iqbal 15 min read

01
Executive Summary

This document outlines the successful migration of on-premises servers to AWS in the mobilize phase. An AWS Organization structure was established with a "development" account in sandbox OU to manage the migration phase. The AWS infrastructure was built in the us-east-1 region across three Availability Zones. AWS Application Migration Service (MGN) was used for the replication of multiple domain-joined and standalone servers, with custom launch templates making sure IP addresses and security configurations remains same as they were in on premises.

02
AWS Organization and Account

We created a separate OU called "Sandbox" for the Mobilize phase, and set up a dedicated account for the migration phase. This account was used throughout the entire Mobilize phase.

03
AWS Infrastructure

04
Networking Setup

A Virtual Private Cloud (VPC) was set up to align with the existing on-premises network design and to support the migrated workloads. The VPC for this migration environment is named Xxx-Mobilize.

4.1 Internet and NAT Gateway

An Internet Gateway and a NAT Gateway were deployed to provide outbound internet access for resources in public and private subnets, respectively. A public subnet named Xxx-PublicSubnet1-1a was defined with the CIDR block 10.0.0.0/24.

4.2 DHCP Configuration and Subnet Planning

To minimize network configuration changes during the server migration to AWS, the subnet CIDR blocks in AWS were matched with those of the on-premises environment i.e 10.0.0.0/24. This setup allowed the migrated servers to keep their original IP addresses.

05
Server Replication with AWS Application Migration Service

5.1 AWS Application Migration Service (MGN)

Application Migration Service (MGN) is a fully managed AWS service that simplifies and automates the migration of physical, virtual, or cloud-based servers to AWS. It enables continuous data replication from source servers to AWS, allowing for minimal downtime during cutover.

5.2 Setup Process

1

IAM Role and STS Configuration

An AWS Identity and Access Management (IAM) role was created with a Secure Token Service (STS) Assume Role policy.

2

Initialization and Replication

After the agent was initialized, the initial sync process began. Nearly 10 terabytes of data were transferred and synchronized.

5.4 Custom Launch Template

During the mobilization phase, custom launch templates were configured with:

  • The VPC and subnet (xxx-PublicSubnet1-1a)
  • The original IP address from the on-premises environment
  • The appropriate security group for the server

06
RD Gateway Configuration

6.1 Overview

In the AWS environment, a Remote Desktop Gateway (RD Gateway) was set up to enable secure RDP access to the migrated servers. This configuration ensures that private servers remain behind the RD Gateway, allowing secure access.

6.2 SSL Certificate Configuration

To secure the RD Gateway, a self-signed certificate was created using PowerShell:

PowerShell certificate-creation.ps1 
New-SelfSignedCertificate -DnsName "ec2-54-205-243-231.compute-1.amazonaws.com" -CertStoreLocation "Cert:\LocalMachine\My"

07
Conclusion

The migration from on-premises to AWS was successfully completed with minimal downtime and maintained security standards. The use of AWS Application Migration Service (MGN) and careful planning of network configurations ensured a smooth transition for all migrated workloads.
